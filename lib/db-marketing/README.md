# @lib/db-marketing

Database-first package for the `MARKETING_DB_URL` database used by the personal
productivity CMS in this repository.

This package owns:

- migration history
- current schema snapshot
- generated TypeScript and JSON schema artifacts
- shared Postgres connection access

This is the canonical home for Notes migration scripts. Root `pnpm run db:*`
commands should proxy into this package rather than duplicating migration logic
inside app packages.

Current tables managed here:

- `user_v1`
- `user_note_v1`
- `user_note_tag_v1`
- `user_note_tag_link_v1`

## Environment

Set:

```bash
export MARKETING_DB_URL="postgres://..."
```

`db:schema:snapshot` and `db:verify` require local PostgreSQL client tools.
Use the same PostgreSQL major version as the target DB server and CI
(`pg_dump`/`psql` 17 for the current workflow setup). The snapshot script fails
fast if the local client major version does not match the server.

## Fresh empty database

Use this flow for a brand-new empty Postgres database:

```bash
pnpm --filter @lib/db-marketing db:migrate
pnpm --filter @lib/db-marketing db:verify
```

What this does:

- creates the baseline tables
- applies all forward migrations
- regenerates schema and type artifacts during verification
- fails if generated artifacts do not match the migrated database

Use the same flow when provisioning a brand-new Notes production database.

## Existing database that already has the baseline schema

Use this only if the database already contains the baseline tables from an
older/manual setup and has not yet been put under migration tracking:

```bash
pnpm --filter @lib/db-marketing db:migrate:baseline
pnpm --filter @lib/db-marketing db:migrate
pnpm --filter @lib/db-marketing db:verify
```

Use `db:migrate:baseline` only for legacy databases that already contain the
baseline schema but were never put under migration tracking. Do not run it as a
normal release step.

## Commands

### Apply migrations

```bash
pnpm --filter @lib/db-marketing db:migrate
```

Run this when the target Notes database should be brought up to the latest
tracked schema. That includes feature-branch development databases and the
production Notes database before or during a coordinated `notes-next` release.

### Verify DB contract

```bash
pnpm --filter @lib/db-marketing db:verify
```

`db:verify` is not read-only. It runs `db:migrate` first, then regenerates
local contract artifacts and checks them with `git diff --exit-code`.

Use `db:verify` mainly on a clean feature branch before merge, or for an
intentional contract check against a target database. It is not required for
every production push.

### Create a new migration

```bash
pnpm --filter @lib/db-marketing db:migration:new -- add_note_status
```

Migration files are forward-only SQL. Do not add `BEGIN` / `COMMIT`; the
runner wraps each file in a transaction.

### Regenerate snapshot and types

Use these when maintaining the package itself:

```bash
pnpm --filter @lib/db-marketing db:schema:snapshot
pnpm --filter @lib/db-marketing db:types:generate
```

Or:

```bash
pnpm --filter @lib/db-marketing db:sync
```

## Files to update when the schema changes

- `migrations/*.sql` — write the new forward migration
- `scripts/verify-contract.mjs` — update structural assertions to match the new schema (add/remove column checks, index checks, constraint checks)
- `schema/current.sql` — regenerated by `db:schema:snapshot`
- `generated/typescript/db-types.ts` — regenerated by `db:types:generate`
- `generated/contracts/db-schema.json` — regenerated by `db:types:generate`
- `generated/contracts/notes-app.json` — regenerated by `app:contract:generate`

Run `pnpm --filter @lib/db-marketing db:sync` to regenerate all derived artifacts
in one step, or `db:verify` to also run the structural checks.

## Migration workflow for feature branches

### Current setup (shared dev database)

All branches share a single dev database. Migrations are forward-only and
applied immediately.

**Safe workflow for feature branches:**

1. Write backwards-compatible migrations when possible (additive columns,
   nullable fields, new tables).
2. Run `db:migrate` on the feature branch to apply changes.
3. Run `db:verify` on the feature branch — this catches drift between the
   migration, verify script, and generated artifacts **before** merging.
4. Merge to main and deploy.

**When a migration is NOT backwards-compatible** (drops columns, renames, etc.):

1. Coordinate: deploy code changes and migration together.
2. If the feature is rejected, write a compensating migration to undo the
   change on a new branch.
3. Accept that on a shared dev DB, destructive migrations carry risk.

## Production release guidance

When Notes production is stale and you need to promote the newest version:

1. Verify code and generated artifacts from the repo root.
2. Point `MARKETING_DB_URL` at the production Notes database.
3. Run `db:migrate`.
4. Deploy `apps/notes-next` on Railway.
5. Run `db:embeddings:regenerate` only if embeddings or search data need repair.

If production is a legacy untracked database that already contains the baseline
schema, run `db:migrate:baseline` once before step 3.

### Future: Railway PR Environments (recommended upgrade path)

Railway supports automatic PR preview environments with isolated databases.
When enabled:

- Each PR gets its own Postgres instance with `MARKETING_DB_URL` injected.
- Migrations run against the isolated DB, not shared dev.
- The PR environment (and its database) is destroyed when the PR closes.
- No risk to the shared dev database from experimental migrations.

To enable: go to Railway dashboard → Project Settings → Environments → enable
PR Environments. See https://docs.railway.com/environments for details.
